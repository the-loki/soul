# 项目协作规则

## 语言与文档

- 项目文档必须使用中文编写。
- 面向开发者的架构说明、设计说明、操作手册和示例说明都应优先放在 `docs/` 下。
- 代码本身不得使用中文，包括源码注释、错误字符串、诊断消息、测试名称和示例代码。
- 代码中的关键不变量、`unsafe` 安全前提、复杂迁移逻辑和跨模块边界必须使用英文注释说明。
- 普通自解释代码不需要为了注释而注释。
- 新增测试项应在测试函数前使用简短英文注释说明覆盖的行为、回归点或边界条件；注释不得重复测试名本身。

## 文档查询

使用 `ctx7` CLI 获取当前文档，只要问题涉及库、框架、SDK、API、CLI 工具或云服务，即使是常见技术也必须查询。

适用范围包括 API 语法、配置、版本迁移、库相关调试、安装和 CLI 用法。

不适用范围包括普通重构、从零写业务脚本、业务逻辑调试、代码审查和通用编程概念。

步骤：

1. 解析库：`npx ctx7@latest library <name> "<用户问题>"`
2. 根据名称匹配、描述相关性、代码片段数量、来源声誉和 benchmark 分数选择最佳 ID。
3. 获取文档：`npx ctx7@latest docs <libraryId> "<用户问题>"`
4. 基于获取到的文档回答或实现。

除非用户直接提供 `/org/project` 格式的库 ID，否则必须先调用 `library`。

## 架构原则

- 模块边界优先清晰、可测试、可替换，再考虑局部性能优化。
- 公共接口应避免泄漏可替换实现的内部细节。
- 设计决策应优先沉淀到 `docs/` 下的设计文档中，不写入协作规则。

## AI 友好与外部操作

- 面向外部工具的操作应稳定、结构化、可审计。
- 外部操作不得绕过模块边界破坏内部不变量。
- 外部操作失败时应提供明确错误码和英文诊断信息；中文解释放在文档或后续本地化资源中，不写入源码。
- 能力发现、权限控制和命令日志等机制应保持可复用、可测试。

## `unsafe` 规则

- 外部 API 默认保持 safe Rust。
- `unsafe` 只能出现在明确需要的内部模块或 FFI 边界。
- 每个关键 `unsafe` 块必须用英文注释说明安全前提和调用方必须满足的不变量。
- 涉及 `unsafe` 的行为必须有单元测试覆盖。
